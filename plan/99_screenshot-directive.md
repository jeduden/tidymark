# Build directive — declared artifact system

## Goal

Give documentation build artifacts the same
first-class status that `<?catalog?>` and
`<?include?>` give to generated text. Authors
declare the recipe, inputs, and output path.
`mdsmith build` runs the recipe when sources
change. `mdsmith fix` keeps the embedded link
in sync with the params — with no network
access and no external tools.

## Context

This repo faces the same problem in two forms:
browser screenshots for feature docs
(manual today) and `demo.gif` generated by
VHS from `demo.tape` (also manual today). The
pattern is straightforward: sources change,
artifact must be regenerated.

[Make][make] calls this a rule with dependencies
and a recipe. `mdsmith build` applies the same
model to documentation artifacts: declare the
output, the tool, and the inputs — the system
handles the rest.

[make]: https://www.gnu.org/software/make/

### Why split lint from build

Running a recipe requires an external tool,
possibly a network target, and time. Folding
that into `mdsmith fix` would make the linter
slow and stateful.

- **Lint layer (always on).** MDS039 validates
  params and rewrites the body link on
  `mdsmith fix`. No external tools needed.
- **Build layer (opt-in).** `mdsmith build`
  dispatches to the recipe driver. It is
  never a side effect of `check`/`fix`.

[gensection]: ../docs/background/archetypes/generated-section/README.md

## Design

### Directive syntax

```text
<?build
recipe: screenshot
url: /inbox
output: docs/inbox.png
?>
![Inbox screenshot](docs/inbox.png)
<?/build?>
```

```text
<?build
recipe: vhs
input: demo.tape
output: demo.gif
?>
![Terminal demo](demo.gif)
<?/build?>
```

Common parameters (all recipes):

| Name     | Required | Description                                 |
|----------|----------|---------------------------------------------|
| `recipe` | yes      | Registered recipe name                      |
| `output` | yes      | Artifact path relative to the Markdown file |

Recipe-specific parameters are defined by each
recipe (built-in or declared in config).

### Output type

`output` can be any file: PNG, GIF, WebP, MP4,
SVG, Markdown, JSON, or any other type. The
recipe declares what it produces; MDS039
applies no extension filter. The generated
body is controlled by the recipe's
`body_template` (see below).

### Generated body (lint layer)

Each recipe declares a `body_template`. The
rendered template replaces the directive body
on `mdsmith fix`. Built-in defaults:

| Recipe       | Default `body_template` |
|--------------|-------------------------|
| `screenshot` | `![{alt}]({output})`    |
| `vhs`        | `![{alt}]({output})`    |
| custom       | `[{output}]({output})`  |

When `alt` is absent, it is derived as
`"{recipe} output: {output}"` so the body
always passes MDS032. Users can set `alt:` in
the directive to override.

MDS039 compares the rendered template to the
actual body and reports a stale-section
diagnostic. `Fix` rewrites the body. Reuses
`gensection.Engine` unchanged.

### Built-in recipes

**`screenshot`** — captures a URL via
[chromedp][chromedp]:

| Param      | Required | Default    |
|------------|----------|------------|
| `url`      | yes      | —          |
| `selector` | no       | full page  |
| `viewport` | no       | `1280x800` |
| `wait`     | no       | `0` ms     |
| `click`    | no       | —          |
| `hide`     | no       | `[]`       |

**`vhs`** — renders a `.tape` file:

| Param   | Required |
|---------|----------|
| `input` | yes      |

[chromedp]: https://github.com/chromedp/chromedp
[vhs]: https://github.com/charmbracelet/vhs

### User-defined recipes

Recipes are declared in `.mdsmith.yml`. Any
output type is allowed:

```yaml
build:
  recipes:
    mermaid:
      command: "mmdc -i {input} -o {output}"
      body_template: "![{alt}]({output})"
      params:
        required: [input]
        optional: [theme]
    api-spec:
      command: "redocly bundle {input} -o {output}"
      body_template: "[API spec]({output})"
      params:
        required: [input]
```

`{param}` tokens expand to directive parameter
values at build time. MDS039 validates that
`required` params are present.

### Rule: MDS039 (build)

- ID: `MDS039`
- Name: `build`
- Category: `meta`
- Default: enabled
- Fixable: yes (lint layer only)

Validation:

- `recipe` is present and resolves to a known
  recipe (built-in or declared in config)
- `output` is a relative path inside the
  project root, no `..`
- Recipe-specific required params are present

### `mdsmith build` subcommand

```text
mdsmith build [paths...] [flags]
```

Flags:

- `--recipe NAME` — limit to one recipe
- `--base-url URL` — prefix for path-only URLs
- `--dry-run` — list targets; write nothing
- `--timeout DURATION` — per-artifact timeout
  (default `30s`)

Behaviour:

1. Walk files the same way `check`/`fix` do.
2. Collect and validate `<?build?>` blocks.
3. Dispatch each block to its recipe driver in
   declaration order.
4. Write artifacts atomically to `output`.
5. Exit non-zero on any failure; print a
   per-file `OK | FAIL` summary.

### Configuration

```yaml
build:
  base-url: ""    # joined to path-only URLs
  recipes: {}     # user-defined recipe declarations
```

Per-directive params override config defaults.

### Interaction with existing rules

- **MDS032**: derived alt text is non-empty.
- **MDS027**: a missing artifact fires MDS027
  and points the user at `mdsmith build`.
- **`merge-driver`**: regenerates the body on
  conflict; artifact bytes are not regenerated.

## Tasks

1. Define a `Builder` interface and recipe
   registry in `internal/build/`. Built-in
   impls: `screenshot` (chromedp) and `vhs`
   (exec). Custom-recipe impl reads the command
   template from config and runs via `os/exec`.
2. Create the `<?build?>` directive in
   `internal/rules/build/` using
   `gensection.Engine`. Register as MDS039,
   category `meta`. `Generate` renders the body
   template; it never calls a builder.
3. Add MDS039 fixtures under
   `internal/rules/MDS039-build/`:
   `good/`, `bad/`, `fixed/`. Cover built-in
   recipes and a user-defined recipe.
4. Wire the rule into `cmd/mdsmith/main.go`.
5. Add `mdsmith build` subcommand. Reuse the
   file walker and AST parser. Wire all flags.
6. Add the `build:` config block in
   `internal/config/`. Surface the recipe
   registry to MDS039 and the build command.
7. Integration test: `screenshot` against
   `httptest.Server` writes a non-empty PNG.
   A `cp`-based custom recipe writes any file.
   Both skip without their prerequisite binary.
8. Document MDS039 README and a user guide at
   `docs/guides/directives/build.md`. Update
   the gensection archetype doc.
9. Add a demo using a static HTML file so it
   runs without a dev server.

## Acceptance Criteria

- [ ] `<?build recipe:screenshot url:... output:...?>`
      regenerates its body on `mdsmith fix`
- [ ] `<?build recipe:vhs input:demo.tape output:demo.gif?>`
      regenerates its body on `mdsmith fix`
- [ ] MDS039 reports stale-section when the body
      diverges from the rendered `body_template`
- [ ] MDS039 rejects unknown recipe, missing
      `output`, path traversal, and missing
      required params
- [ ] A user-defined recipe with a `cp` command
      produces its output file
- [ ] `output` can be any file extension; no
      extension filter is applied by MDS039
- [ ] `mdsmith build` against `httptest.Server`
      writes a non-empty PNG
- [ ] `mdsmith build --dry-run` lists every
      target without running any tool
- [ ] `mdsmith build` exits non-zero on failure
      with a per-file `OK | FAIL` summary
- [ ] `mdsmith check` does **not** run any
      external tool for `<?build?>` blocks
- [ ] `build:` config defaults apply to
      directives and are overridden per-directive
- [ ] CI without chromium still passes; build
      tests are skipped, not failed
- [ ] Merge driver regenerates `<?build?>`
      bodies on conflict (gensection engine)
- [ ] All tests pass: `go test ./...`
- [ ] `go tool golangci-lint run` reports
      no issues
